# ChatRoom 集約と Membership 集約雑考

お題: ChatRoom 集約と Membership 集約はどのように関係すべきか？
要件: チャットルームが作成されたら、作成したユーザーは必ずそのチャットルームのメンバーになる

## パターン1: ChatRoom 集約配下のエンティティとして Members を持つ場合

### メリット

- 不変条件を守りやすい
  - 「チャットルームが作成されたら、作成したユーザーは必ずそのチャットルームのメンバーになる」のような要件を自然に定義できる
- 純粋に実装しやすい
  - チャットルームがメンバーコレクションを保持するというのはメンタルモデル的にもわかりやすい

### デメリット


## パターン2: ChatRoom 集約と Membership 集約を分離するが、同一トランザクションでコミットする

### メリット

- 強い整合性を担保できる
  - もし途中でプロセスが死んでもロールバックが発生するので不整合が起こる心配がない
- 実装は楽
  - 雑にアプリケーションサービスでトランザクションを張って2回書き込んでコミットすれば良い

### デメリット

- スケールしない
  - トランザクションの保持期間が長くなるので、DB負荷が容易に上がる
- DDD の原則にも反する
  - DDD 的には集約はトランザクションの単位である
  - 同一トランザクションの中で2つの集約がコミットされるのはその原則に違反する
  - こういうことをやるとアプリケーションサービスがどんどん肥大化し、最終的にはコードベース全体がスクリプト化する

## パターン3: ChatRoom 集約と Membership 集約を分離し、別トランザクションでコミットする


### デメリット

- 2つのトランザクションの間でプロセスが死ぬと結果整合性すら担保されなくなる


## パターン4: ChatRoom 集約と Membership 集約を分離し、Membership 集約は ChatRoom 集約のイベントを購読する

### メリット

- 集約単位でトランザクションを張れる
  - 例えばメンバーの大量削除のような大規模リクエストが来ても、チャットルームに対する操作は影響を受けずに済む
  - DDD の集約はトランザクション単位という原則も守ることができる
- 結果整合性を担保できる
  - Membership 集約が適切にイベントを吸ってさえいれば、いつかは整合性が担保される


### デメリット

- fan-in の問題が起こる
  - Membership 集約は ChatRoom 集約のイベントも吸わなければいけないし、エンドユーザーからのコマンド (メンバーの削除や編集) も受け取らなければならない
  - それらの順序はどのように担保すれば良い？
    - …と書いていて思ったが、Membership 集約に到達した順序を正とすれば良いのであって、別にそこまで気にすることもないのではないか？
    - Membership 集約は fan-in でデータを (コマンドやイベントとして) 吸い出し、順序付けられたイベントを吐くだけ
